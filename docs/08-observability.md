# JCarranza Mini-Midoffice — Observability

> Phase 8 of 9. Logback, three separate log categories, MDC requestId, rotation and retention.

---

## 1. Context: How the Team Works with Logs

In mid-office systems of this kind, 2nd-level support connects via SSH to the server and reviews log files to diagnose what happened. No Kibana dashboards, no distributed traces: just `cat`, `grep` and `tail -f`. The logging system is designed for exactly that scenario.

The question guiding the design is: **can a support technician, with only grep and less than 2 minutes, understand what happened in a booking that failed?**

---

## 2. Three Log Categories and Their Files

```
/usr/local/tomcat/logs/mini-midoffice/
├── business.log          — business events with IDs always present
├── sabre-integration.log — GDS calls (endpoint, duration, errors)
└── errors.log            — unhandled exceptions with full stack trace
```

Each file has a different audience and purpose:

| File | Audience | Minimum Level | When to Check |
|---|---|---|---|
| `business.log` | Business support, operations | INFO | "When was this booking made? Who cancelled it?" |
| `sabre-integration.log` | Technical support, GDS integration | INFO/WARN/ERROR | "What did Sabre return? How long did it take?" |
| `errors.log` | Developers, critical incidents | ERROR | "Why are there 500s? What exception?" |

## 3. MDC requestId — The Key to Cross-File Diagnosis

Each HTTP request receives a unique `requestId` (UUID truncated to 8 hex chars) generated by `RequestIdFilter` and stored in the SLF4J MDC.

**The requestId appears in every log line for that request**, across all files. To reconstruct the complete trace of request `c1849d39`:

```bash
grep c1849d39 logs/mini-midoffice/*.log
```

Result:
```
business.log:          2026-06-18 21:07:48.089 ... [c1849d39] ... FLIGHT_CHECK_START travellerId=1 flightKey=LX|23|JFK|GVA|...
business.log:          2026-06-18 21:07:48.771 ... [c1849d39] ... BOOKING_RESERVED bookingId=2113 travellerId=1 ...
sabre-integration.log: 2026-06-18 21:07:48.089 ... [c1849d39] ... Sabre flightCheck LX23 JFK/GVA on 2026-07-18
sabre-integration.log: 2026-06-18 21:07:48.100 ... [c1849d39] ... SABRE_REQUEST → POST /v1/offers/flightCheck
sabre-integration.log: 2026-06-18 21:07:48.736 ... [c1849d39] ... SABRE_RESPONSE ← POST /v1/offers/flightCheck HTTP 200 (635ms)
sabre-integration.log: 2026-06-18 21:07:48.750 ... [c1849d39] ... Sabre flightCheck available=true price=3834.22 EUR
```

In ~2 seconds the technician knows: who booked, what flight, what Sabre returned, how long it took, and at what price it was confirmed.

## 4. Log Format

```
%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level [%X{requestId:-no-rid}] %logger{36} - %msg%n
```

Real example:
```
2026-06-18 21:07:48.736 [http-nio-8080-exec-3] INFO  [c1849d39] c.m.i.sabre.client.SabreHttpClient - SABRE_RESPONSE ← POST /v1/offers/flightCheck HTTP 200 (635ms)
```

Fields:
- `2026-06-18 21:07:48.736` — timestamp with milliseconds (needed to measure durations)
- `[http-nio-8080-exec-3]` — Tomcat thread (to understand concurrency)
- `INFO` — level with fixed padding (5 chars) for visual alignment
- `[c1849d39]` — requestId from MDC
- `c.m.i.sabre.client.SabreHttpClient` — abbreviated class (36 chars max)
- `SABRE_RESPONSE ← POST ...` — readable message with category prefix

## 5. Logging System Architecture

```
RequestIdFilter (web.xml, first filter)
    │  UUID → MDC["requestId"]
    │
    ├─► business.log    ← logger "com.jcarranza.minimidoffice.business"
    │       ProfileServiceImpl   → PROFILE_CREATED, PROFILE_UPDATED, PROFILE_DELETED
    │       BookingServiceImpl   → FLIGHT_CHECK_START, BOOKING_RESERVED, BOOKING_CANCELLED
    │       SearchServiceImpl    → FLIGHT_SEARCH, FLIGHT_SEARCH_RESULT
    │
    ├─► sabre-integration.log   ← logger "com.jcarranza.minimidoffice.integration.sabre"
    │       (all loggers under that package are automatically routed here)
    │       SabreFlightSearchAdapter → call and result
    │       SabreFlightCheckAdapter  → call and result
    │       SabreHttpClient          → SABRE_REQUEST, SABRE_RESPONSE (with duration ms)
    │       SabreFlightCheckMapper   → SABRE_BUSINESS_ERROR (WARN), SABRE_NO_OFFERS (WARN)
    │       SabreOAuthProvider       → token refresh
    │
    └─► errors.log      ← logger "com.jcarranza.minimidoffice.errors"
            GlobalExceptionHandler → UNHANDLED_EXCEPTION (only 500s, full stack trace)
```

### Routing Without Changing Adapter Code

The Sabre adapters (`SabreFlightSearchAdapter`, `SabreFlightCheckAdapter`, `SabreHttpClient`, `SabreFlightCheckMapper`) use `LoggerFactory.getLogger(getClass())` — their logger names are `com.jcarranza.minimidoffice.integration.sabre.*`. In Logback, the logger `com.jcarranza.minimidoffice.integration.sabre` automatically captures that entire hierarchy with `additivity=false`, routing all its messages to `sabre-integration.log` without touching the adapter code.

## 6. Business Events in `business.log`

All business events have uppercase prefixes and always include the main business ID:

| Event | Class | Required Fields |
|---|---|---|
| `PROFILE_CREATED` | `ProfileServiceImpl` | `travellerId`, `email`, `company` |
| `PROFILE_UPDATED` | `ProfileServiceImpl` | `travellerId` |
| `PROFILE_DELETED` | `ProfileServiceImpl` | `travellerId` |
| `FLIGHT_SEARCH` | `SearchServiceImpl` | `origin`, `dest`, `from`, `to`, `los`, `pax` |
| `FLIGHT_SEARCH_RESULT` | `SearchServiceImpl` | `origin`, `dest`, `results` |
| `FLIGHT_CHECK_START` | `BookingServiceImpl` | `travellerId`, `flightKey` |
| `BOOKING_RESERVED` | `BookingServiceImpl` | `bookingId`, `travellerId`, `flight`, `confirmedPrice`, `currency` |
| `BOOKING_CANCELLED` | `BookingServiceImpl` | `bookingId`, `travellerId` |

Business IDs are always present so you can grep by booking or by traveller:

```bash
grep "travellerId=42"  business.log  # all activity for a traveller
grep "bookingId=2113"  business.log  # complete history of a booking
```

## 7. Sabre Integration Logs in `sabre-integration.log`

### Successful Call (INFO)

```
2026-06-18 21:07:48.100 [http-nio-8080-exec-3] INFO  [c1849d39] c.m.i.sabre.client.SabreHttpClient - SABRE_REQUEST → POST /v1/offers/flightCheck
2026-06-18 21:07:48.736 [http-nio-8080-exec-3] INFO  [c1849d39] c.m.i.sabre.client.SabreHttpClient - SABRE_RESPONSE ← POST /v1/offers/flightCheck HTTP 200 (635ms)
```

The duration in milliseconds allows detecting GDS degradation before it impacts the user.

### Sabre Business Error (WARN)

Sabre returns errors as JSON objects `{"category":"...", "type":"...", "description":"..."}`, **not as strings**. This was the real bug that caused `JsonMappingException` and 504s until `SabreError` was added as an inner class. The WARN captures the three fields separately:

```
2026-06-18 21:17:40.844 [http-nio-8080-exec-5] WARN  [9aa54ae3] c.m.i.s.m.SabreFlightCheckMapper - SABRE_BUSINESS_ERROR category=Revalidation error type=Itinerary could not be priced. description=No such flight (XX 9999)
```

With this, support can distinguish between:
- `description=No such flight` → flight does not exist in the cert dataset, or the date is in the past
- `description=Itinerary could not be priced.` → incorrect PCC, incorrect flight times
- `description=Wrong clientID or clientSecret` → expired credentials

### HTTP Error Response (ERROR)

```
2026-06-18 xx:xx:xx.xxx [thread] ERROR [requestId] c.m.i.sabre.client.SabreHttpClient - SABRE_ERROR ← POST /v1/offers/flightCheck HTTP 401 (83ms) body={"message":"Credentials are missing..."}
```

### Network Error/Timeout (ERROR)

```
2026-06-18 xx:xx:xx.xxx [thread] ERROR [requestId] c.m.i.sabre.client.SabreHttpClient - SABRE_IO_ERROR ← POST /v1/offers/flightCheck (30001ms): Read timed out
java.net.SocketTimeoutException: Read timed out
    at ...
```

## 8. Unhandled Exceptions in `errors.log`

Only cases that `GlobalExceptionHandler` does not know how to handle specifically (HTTP 500) reach here. `BusinessException` (400/409) and `IntegrationException` (502/504) **do not go to `errors.log`** — they have their own handling and are not bugs.

Example of what an unexpected `NullPointerException` in a controller would produce:

```
2026-06-18 xx:xx:xx.xxx [http-nio-8080-exec-7] ERROR [f3a91b20] com.jcarranza.minimidoffice.errors - UNHANDLED_EXCEPTION on POST /api/bookings: null
java.lang.NullPointerException: Cannot invoke "String.trim()" because "this.email" is null
    at com.jcarranza.minimidoffice.service.ProfileServiceImpl.create(ProfileServiceImpl.java:43)
    at com.jcarranza.minimidoffice.service.ProfileServiceImpl$$FastClassBySpringCGLIB$$...invoke(...)
    at org.springframework.aop.framework.CglibAopProxy$CglibMethodInvocation.proceed(...)
    ...
```

The requestId `f3a91b20` allows searching in the other logs for what that request was doing just before it exploded.

## 9. Rotation and Retention Policy

| File | Rotation | Max Size/File | Retention | Reason |
|---|---|---|---|---|
| `business.log` | Daily | 50 MB | 30 days | Support queries rarely go back more than a month |
| `sabre-integration.log` | Daily | 100 MB | 60 days | GDS integrations can have failure patterns that emerge over weeks |
| `errors.log` | Daily | — | 90 days | Silent bugs may not be reported until weeks later |

Rotated files are compressed with `.gz` to reduce disk space.

The property `${catalina.base:-/tmp}/logs/mini-midoffice` resolves to:
- `/usr/local/tomcat/logs/mini-midoffice` when running in Tomcat
- `/tmp/logs/mini-midoffice` in tests/development without Tomcat

## 10. How to Diagnose Common Scenarios

### "A client says their booking failed — what happened?"

```bash
# Search by bookingId if known
grep "bookingId=2113" business.log

# Search by travellerId
grep "travellerId=42" business.log

# If the booking never reached DB, search by flightKey
grep "LX|23|JFK|GVA" sabre-integration.log
```

### "FlightCheck is taking more than 3 seconds — is there degradation in Sabre?"

```bash
grep "SABRE_RESPONSE.*flightCheck" sabre-integration.log | grep -v "([0-2][0-9][0-9]ms)" | tail -20
# Shows all flightCheck responses that took ≥ 300ms
```

### "We are getting 500s in production — what exception is it?"

```bash
tail -200 errors.log
# or search by approximate timestamp:
grep "2026-06-18 21:" errors.log
```

### "When was the last time Sabre returned a business error?"

```bash
grep "SABRE_BUSINESS_ERROR" sabre-integration.log | tail -5
```

### "Sabre is responding 401 — did the token expire?"

```bash
grep "SABRE_ERROR.*401\|token.*expired\|SABRE_ERROR.*credential" sabre-integration.log | tail -10
```

## 12. Design Decisions

| Decision | Alternative | Reason |
|---|---|---|
| Three separate physical files | A single `app.log` | Support can `cat business.log` without Sabre noise; integration can `tail -f sabre-integration.log` without business spam |
| Named logger `com.jcarranza.minimidoffice.business` | Separate appender in each service class | All business events with a single `grep`; new services join in one line (`LoggerFactory.getLogger("com.jcarranza.minimidoffice.business")`) |
| Logger `com.jcarranza.minimidoffice.integration.sabre` routes the entire package hierarchy | Separate logger per adapter | Adapters don't need to change — routing is by package, not by class |
| `additivity=false` on the three named loggers | `additivity=true` (default) | Prevents business messages from appearing duplicated in the root logger (console) |
| `errors.log` only for 500s | Also 4xx errors | 400/409 are expected user errors, not bugs. Mixing them with 500s would make diagnosis harder |
| 8-char requestId (truncated UUID) | Full UUID (36 chars) | 8 chars is sufficient entropy to distinguish requests in a day's log; the log remains much more readable |
| Timing in `SabreHttpClient.execute()` | Timing in the adapters | The HTTP client is the only point where actual network time is measured, without serialization overhead |
| `SABRE_BUSINESS_ERROR` with separate fields | `toString()` of the error | `grep "description=No such flight"` works; `grep "Revalidation error / No such flight"` requires knowing the toString format |
